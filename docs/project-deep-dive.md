# 项目详细说明

## 1. 这个项目是做什么的

智能工单平台是一个面向服务台场景的后端项目。它的核心目标不是只实现“提工单、查工单”这样简单的 CRUD，而是把一个真实服务台系统中更有代表性的能力放到一个项目里统一展示，包括：

- 工单主流程
- 权限与角色控制
- SLA 规则与违约扫描
- 自动分派
- 审批流
- 多类型工单
- 多视角摘要
- Agent 对话入口
- RAG 历史经验检索

从定位上看，这个项目更像一个“高完成度、可讲清楚的工程化 MVP”，而不是只做概念演示的 Demo。它保留了较强的业务复杂度，也保留了面向扩展的结构设计，因此适合用于：

- Java 后端实习 / 校招简历项目
- Agent 应用方向的实习展示
- 面试时讲架构、分层、业务抽象与工程取舍

## 2. 为什么采用模块化单体

这个项目没有一开始就拆成微服务，而是采用了“模块化单体”的方式组织代码。这样做有两个现实原因：

1. 对个人项目和简历项目来说，模块化单体比微服务更容易把精力放在业务抽象和工程质量上，而不是把大量时间消耗在服务治理、部署编排和分布式基础设施上。
2. 服务台类系统的很多能力天然强关联，例如工单、审批、分派、SLA、Agent、RAG，它们在领域上有大量共享上下文。先做模块化单体，既能保留清晰边界，也能降低跨服务通信和拆分成本。

这里的关键点不是“没有拆服务”，而是“虽然还是单进程应用，但内部已经有清晰的模块边界和职责分层”。

## 3. 项目的整体结构

项目根目录下的核心模块如下：

```text
smart-ticket-platform
|- smart-ticket-app
|- smart-ticket-common
|- smart-ticket-domain
|- smart-ticket-infra
|- smart-ticket-auth
|- smart-ticket-biz
|- smart-ticket-rag
|- smart-ticket-agent
`- smart-ticket-api
```

可以把它理解成四层：

- 基础层：`common`、`domain`
- 基础设施适配层：`infra`、`auth`
- 业务能力层：`biz`、`rag`、`agent`
- 入口层：`api`、`app`

这套结构的目的，是让“领域对象、业务规则、外部接入、HTTP 接口、Agent 编排”分开，而不是全部堆进 controller 或 service。

## 4. 各模块职责详解

### 4.1 `smart-ticket-app`

这个模块是应用启动模块，负责把其他模块装配成一个可运行的 Spring Boot 应用。

它主要承担三件事：

- 启动 Spring Boot
- 配置组件扫描
- 开启定时任务能力

这里不会承载业务逻辑。它更像整个项目的装配入口。

### 4.2 `smart-ticket-common`

这个模块放项目里最通用、最稳定的东西，例如：

- 统一响应结构
- 分页结果结构
- 业务异常
- 错误码定义

它的意义在于让整个项目的错误处理和接口返回方式保持一致。  
例如 controller 不需要各自约定失败格式，service 不需要随意抛字符串异常，而是统一使用业务错误码。

### 4.3 `smart-ticket-domain`

这是领域数据模型层，放的是“业务对象长什么样”和“数据库映射怎么定义”。

主要包括：

- 实体类，例如 `Ticket`、`TicketApproval`、`TicketSlaPolicy`
- 枚举类，例如工单状态、工单类型、优先级、审批状态
- MyBatis Mapper 定义
- Mapper XML

这个模块不负责业务流程编排，它只负责表达“有哪些领域对象”和“如何持久化”。

### 4.4 `smart-ticket-infra`

这个模块负责基础设施接入，重点是对外部能力的适配，而不是业务规则本身。

当前主要涉及：

- Redis
- Spring AI
- 向量存储接入

它的作用是让业务层不必直接耦合底层客户端细节。  
例如 RAG 检索服务只需要关注“怎么检索”，不需要在业务代码里到处写具体连接和底层配置。

### 4.5 `smart-ticket-auth`

这个模块集中处理认证和安全配置。

主要职责：

- JWT 生成与校验
- Spring Security 配置
- 登录用户加载
- 未认证 / 无权限请求的统一处理

这个模块只解决“你是谁、你有没有系统级角色权限”。  
它不负责工单层面的关系权限，例如“某人能不能看这张工单”“某人能不能转派这张工单”，这些逻辑放在 `biz` 里。

这是一个很重要的边界：  
`auth` 处理系统安全，`biz` 处理领域权限。

### 4.6 `smart-ticket-biz`

这是项目最核心的模块，绝大多数业务规则都在这里。

它承载的能力包括：

- 工单创建与主流程流转
- 工单查询与详情装配
- 评论与操作日志
- SLA 策略与实例
- 自动分派
- 审批流
- 多类型工单规则
- 多视角摘要

从代码组织上，这个模块不是单一一个 `TicketService` 大而全到底，而是逐步拆成了更细的服务：

- `TicketCommandService`
- `TicketQueryService`
- `TicketWorkflowService`
- `TicketCommentService`
- `TicketQueueBindingService`
- `TicketSummaryService`
- `TicketApprovalService`
- `TicketAssignmentRuleService`
- `TicketSlaService`

同时又保留了一个 `TicketService` 作为门面，对上层暴露相对稳定的业务入口。

这种结构的好处是：

- API 层不需要知道太多内部实现细节
- 单个 service 的职责更清晰
- 更适合后续补测试
- 后续真的要拆服务时，边界已经比较清楚

### 4.7 `smart-ticket-rag`

这个模块负责历史经验检索能力。

它不是一个泛化的知识库，而是围绕“历史工单经验检索”做的 RAG 子系统。主要能力包括：

- 工单关闭后的知识构建
- Embedding 生成
- query rewrite
- rerank
- 检索主路径 / fallback 切换

当前检索策略有两条路：

1. PGvector 主路径  
当环境具备 PostgreSQL + pgvector 时，优先走向量检索。

2. MySQL fallback  
当主路径不可用，或者开发环境没有准备完整向量库时，退回 MySQL 中存储的向量数据与本地计算逻辑。

这个设计很实用，因为它平衡了两个目标：

- 演示时要体现“项目有标准向量检索主路径”
- 开发时又不能因为没配好外部依赖而完全跑不起来

### 4.8 `smart-ticket-agent`

这个模块负责自然语言入口。

它不是一个“完全自由对话”的聊天模块，而是围绕工单系统内部能力做的 Agent 编排层。核心思路是：

- 先做意图识别
- 再做工具调用
- 再做会话上下文处理
- 必要时做多轮澄清

当前支持的主要意图包括：

- 查询工单
- 创建工单
- 转派工单
- 搜索历史经验

这个模块的几个关键点：

#### 1. 规则路由优先

项目没有把所有意图识别都完全交给大模型，而是先做规则路由，再决定是否进入工具链。  
这样更稳定，也更适合对固定业务场景做可控实现。

#### 2. Tool Calling 是业务边界

Agent 不直接写数据库，不直接绕过业务服务。  
所有执行动作最终都通过工具，再进入 `biz` 层完成。这保证了：

- 权限规则不会被绕过
- 日志与审计仍然统一
- Agent 不会成为“第二套业务入口”

#### 3. 多轮澄清

例如创建工单时，如果缺少标题、描述、分类或优先级，Agent 不会盲目猜，而是保存待补参数和草稿状态，继续和用户对话，直到信息足够。

这让 Agent 功能不只是“把自然语言直接映射成一次请求”，而是有了更接近真实使用场景的交互能力。

### 4.9 `smart-ticket-api`

这个模块是 HTTP 接口层。

主要职责：

- Controller
- DTO / VO
- assembler
- 请求解析与参数转换
- 接口层异常处理

它的工作重点是协议适配，而不是业务处理。  
例如：

- 从请求体解析 DTO
- 把字符串参数转成枚举
- 从认证对象中提取当前用户
- 把业务对象组装为返回 VO

这里尽量避免直接写复杂业务逻辑，让 controller 保持轻量。

## 5. 工单主流程是怎么跑通的

项目里最基础的一条业务线，是工单生命周期。

一个典型流程如下：

1. 用户创建工单
2. 系统写入工单并生成工单号
3. 系统根据类型应用默认分类和优先级
4. 系统记录创建日志
5. 系统生成或刷新 SLA 实例
6. 后续由管理员分派，或队列成员认领
7. 处理人推进状态到 `PROCESSING`、`RESOLVED`
8. 提单人或管理员关闭工单
9. 工单关闭后触发知识构建事件

### 5.1 创建阶段

创建工单时，不只是简单落一条 `ticket` 记录。

当前做了这些规则：

- 支持幂等键
- 支持默认类型
- 按类型自动兜底默认分类与优先级
- 按类型做关键字或结构字段校验
- 保存 `typeProfile`
- 生成操作日志
- 生成 SLA 实例

也就是说，创建工单已经是一个“业务动作”，而不是普通插入。

### 5.2 状态流转阶段

工单状态不是任意跳转，而是受控流转：

```text
PENDING_ASSIGN -> PROCESSING -> RESOLVED -> CLOSED
```

并且每一步都有额外限制。例如：

- 进入 `PROCESSING` 前通常需要处理人
- 审批型工单在审批未通过前不能继续推进
- 关闭必须走单独接口，而不是普通状态更新接口

这些限制都不是前端校验，而是在业务层强制执行。

## 6. 权限是怎么分层设计的

这个项目的权限不是单层的。

### 第一层：系统角色权限

由 `auth` 模块处理。

角色主要包括：

- `USER`
- `STAFF`
- `ADMIN`

这层解决的是“有没有管理员权限”“是不是处理人员”这种系统级问题。

### 第二层：工单关系权限

由 `biz` 模块处理。

例如：

- 当前用户是否可以查看该工单
- 当前用户是否可以转派
- 当前用户是否可以关闭
- 当前用户是否可以认领某个队列下的工单

这样做的好处是，权限判断更贴近业务语义。  
一个人即使登录成功、有 `STAFF` 角色，也不代表他对所有工单都有相同操作权限。

## 7. SLA 是怎么实现的

SLA 这一块是项目里比较有代表性的“规则驱动能力”。

### 7.1 策略层

SLA 策略按工单分类和优先级匹配，定义：

- 首次响应时限
- 解决时限
- 是否启用

### 7.2 实例层

当工单创建或刷新时，会为该工单生成一条 SLA 实例，保存：

- 首响截止时间
- 解决截止时间
- 是否已违约

### 7.3 扫描层

通过定时任务扫描 SLA 实例，判断是否违约。

### 7.4 违约处理

当前第一版实现里，违约后会做：

- 记录审计日志
- 必要时升级优先级
- 某些场景下分配给管理员兜底
- 调用通知服务

这里需要注意：  
通知服务现在还是日志型占位实现，不是完整的站内信 / 邮件 / IM 通道。

## 8. 自动分派是怎么做的

自动分派能力由以下几个概念组成：

- 工单组
- 队列
- 队列成员
- 分派规则

规则本身支持按分类和优先级命中目标，然后再解析出：

- 目标组
- 目标队列
- 目标处理人

如果目标是队列，还会进一步选择：

- 最小负载的可用成员
- 如果没有可用成员，则回退到组负责人

这个设计比“写死给某个人”更接近真实服务台系统。

## 9. 审批流是怎么实现的

项目里的审批流定位为“最小可用审批流”，但已经具备比较完整的骨架。

涉及的对象有：

- 审批模板
- 审批模板步骤
- 审批实例
- 审批实例步骤

### 9.1 哪些工单要审批

当前主要是：

- `ACCESS_REQUEST`
- `CHANGE_REQUEST`

### 9.2 支持的审批方式

- 使用模板审批
- 手动指定审批人

### 9.3 流转规则

- 提交审批
- 当前步骤审批通过
- 激活下一步骤
- 全部通过后标记审批完成
- 任一步驳回后整体驳回

### 9.4 审批与主流程的关系

这是项目里一个很重要的设计点：  
审批不是一个独立孤岛，而是直接约束工单主流程。

审批未通过时，会限制：

- 分配
- 认领
- 转派
- 状态推进
- 关闭

所以审批不是“额外附加功能”，而是业务主流程的一部分。

## 10. 多类型工单为什么重要

如果工单系统只有一种 `INCIDENT`，它更像一个简单问题单系统。  
但在真实服务台场景里，不同类型的工单在信息结构和流程约束上差异很大。

当前支持的类型包括：

- `INCIDENT`
- `ACCESS_REQUEST`
- `ENVIRONMENT_REQUEST`
- `CONSULTATION`
- `CHANGE_REQUEST`

项目通过 `typeProfile` 保存类型差异化字段，并在业务层做对应校验。  
这比把所有字段平铺到主表里更合理，也更适合逐步扩展。

## 11. 多视角摘要为什么值得讲

摘要功能不是简单地把一堆字段拼成一句话，而是按角色视角输出不同重点。

当前支持：

- 提单人视角
- 处理人视角
- 管理员视角

例如同一张工单：

- 提单人更关心进度、下一步和处理人
- 处理人更关心问题现象、最近动作和补充信息
- 管理员更关心风险、阻塞点和是否久未更新

这个功能很适合面试时用来展示“项目不只是流程系统，还开始考虑信息表达和角色场景”。

## 12. Agent 部分的设计思路

这个项目里的 Agent 不是一个“让模型随便发挥”的聊天系统，而是一个受控 Agent。

### 12.1 为什么先做意图路由

因为服务台场景的目标相对明确，用户常见诉求就集中在：

- 查工单
- 建工单
- 转派
- 查历史经验

先做意图路由，可以显著提高稳定性和可控性。

### 12.2 为什么还保留 deterministic fallback

因为纯模型驱动有两个问题：

- 不稳定
- 依赖外部模型环境

所以当前实现是：

- 有条件时尝试走 Spring AI Tool Calling
- 失败或不可用时，退回确定性工具链

这保证了项目在本地和真实演示中都更稳。

### 12.3 为什么要做 pending action

因为创建工单往往一次问不全。  
如果没有 pending action，Agent 只能：

- 要么硬猜
- 要么每次都从头再问

当前实现会把待补字段和已收集草稿写入会话上下文，让多轮创建真正可用。

## 13. RAG 部分的设计思路

RAG 主要服务于一个目标：  
让系统可以基于历史已关闭工单给当前问题提供相似经验参考。

### 13.1 知识从哪里来

知识来源不是外部文档，而是历史工单。  
工单关闭后，会触发知识构建，把有价值的信息整理成知识记录并生成 embedding。

### 13.2 为什么要做 rewrite

因为用户问法往往不稳定。  
例如“登录失败”“测试环境上不去”“账号进不去”，表述不同，但本质相近。

query rewrite 的作用，是把用户表达整理成更适合检索的形式。

### 13.3 为什么要做 rerank

向量检索的初筛结果不一定最适合作为最终展示。  
rerank 用来做轻量排序优化，让结果更稳定。

### 13.4 为什么保留 MySQL fallback

因为很多简历项目的问题是：

- 架构图很好看
- 真要跑时依赖一堆外部资源
- 一旦环境不完整就无法演示

这个项目通过 fallback 解决了这个问题。  
即使没有完整 PGvector 环境，也能保留检索主链的可演示性。

## 14. API 层为什么还单独做 assembler / resolver

很多项目会在 controller 里直接：

- 读请求
- 转换参数
- 调 service
- 手写返回

短期看很快，长期会越来越乱。

这个项目把一些接口层工作拆出来：

- `CurrentUserResolver` 负责统一解析认证用户
- request parser 负责字符串到枚举的转换
- assembler 负责实体 / DTO / VO 的转换

这样 controller 会更轻，职责也更稳定。

## 15. 测试策略是怎么推进的

当前测试重点放在两类：

### 1. Agent / API 入口层测试

验证：

- 未认证访问
- 参数校验
- Agent controller 映射
- Agent 路由后的基本行为

### 2. `biz` 层规则测试

验证：

- 工单创建默认值和类型校验
- 工作流关键约束
- 审批模板规则
- 审批前置限制
- SLA 校验和实例生成
- 自动分派规则统计和校验

这个测试策略是比较务实的：  
先把最容易出错、最能体现业务设计价值的规则固定下来，而不是追求一开始就写满所有集成测试。

## 16. 当前还没有完全做完的部分

虽然项目已经比较完整，但仍有几个边界需要明确：

- SLA 通知通道目前还是日志占位实现
- PGvector 主链路还需要真实环境演示
- Agent 端到端集成测试还可以继续增强
- 摘要策略仍是第一版
- CI 目前是最小版本，只做 `mvn test`

这些内容不影响项目作为简历项目成立，但在面试中应该诚实表达为“已预留并有第一版实现基础”。

## 17. 这个项目最适合怎么讲

如果面试官让你介绍项目，最推荐的讲法不是按文件讲，而是按能力讲：

### 第一部分：先讲定位

“这是一个模块化单体的智能工单平台，重点不是简单 CRUD，而是把工单、审批、SLA、自动分派以及 Agent/RAG 接入做成一套完整后端系统。”

### 第二部分：再讲结构

“我把项目拆成 common、domain、infra、auth、biz、rag、agent、api、app 九个模块，认证、安全、业务规则、检索和接口入口是分开的。”
