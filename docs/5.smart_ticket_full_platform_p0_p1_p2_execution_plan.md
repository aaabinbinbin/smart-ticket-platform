# 完整版智能工单平台 P0 / P1 / P2 可执行任务清单（Spring AI 接入版）

## 1. 文档目的

这份文档用于把当前项目从：

**“工单核心业务系统”**

升级为：

**“具备真实业务复杂度、强 Agent 能力、知识沉淀能力和平台治理能力的完整版智能工单平台”**

本版在原有路线图基础上，新增并细化了：

- 按当前项目结构接入 Spring AI 的详细实施清单
- Spring AI 在 `agent / rag / infra` 三个模块中的职责落位
- Spring AI 与现有 `biz` 模块的边界
- P0 阶段中 Agent、Tool Calling、RAG 的具体接入步骤

---

## 2. 总体原则

### 2.1 不推翻现有项目结构

当前项目已经采用模块化单体结构：

```text
smart-ticket-platform
├─ smart-ticket-app
├─ smart-ticket-common
├─ smart-ticket-auth
├─ smart-ticket-domain
├─ smart-ticket-biz
├─ smart-ticket-agent
├─ smart-ticket-rag
├─ smart-ticket-infra
└─ smart-ticket-api
```

后续接入 Spring AI 时，原则是：

> 只让 Spring AI 接管模型交互层、Tool Calling 层和 RAG 接入层，不重写业务主系统。

### 2.2 保留不动的部分

这些部分继续由你自己的项目代码负责，不因为接入 Spring AI 而重写：

- `smart-ticket-biz`
  - `TicketCommandService`
  - `TicketQueryService`
  - `TicketAssignService`
  - `TicketPermissionService`
  - `TicketKnowledgeService`
- `smart-ticket-auth`
- `smart-ticket-domain`
- `smart-ticket-api` 中除 Agent Controller 外的大部分接口
- 工单状态流转、权限判断、幂等、日志、关闭后知识构建触发

### 2.3 Spring AI 主要接管的部分

Spring AI 主要用于：

- ChatClient
- Tool Calling
- Chat 调用组织
- Advisor
- VectorStore
- EmbeddingModel
- Observability（后续可选）

### 2.4 推荐的总边界

- `biz`：业务真规则层
- `agent`：Spring AI 驱动的对话与 Tool 编排层
- `rag`：知识向量化与检索层
- `infra`：Spring AI 相关模型、Embedding、pgvector、Redis 等适配层

## 3. 按当前项目结构接入 Spring AI 的详细实施清单

这一节是本次新增重点。

### 3.1 模块改造目标

#### `smart-ticket-agent`
目标：从“预留模块”升级为真正的 Agent 接入层。

需要新增或明确这些组件：

- `AgentController`
- `AgentFacade`
- `IntentRouter`
- `AgentSessionService`
- `tool/`
  - `QueryTicketTool`
  - `CreateTicketTool`
  - `TransferTicketTool`
  - `SearchHistoryTool`
- `handler/`
  - `QueryTicketHandler`
  - `CreateTicketHandler`
  - `TransferTicketHandler`
  - `SearchHistoryHandler`
- `prompt/`
  - system prompt
  - per-intent prompt template

#### `smart-ticket-rag`
目标：从“RAG 预留模块”升级为真实检索模块。

需要新增或明确这些组件：

- `EmbeddingService`
- `RetrievalService`
- `QueryRewriteService`
- `TextSplitService`
- `model/`
  - `KnowledgeDocument`
  - `KnowledgeChunk`
  - `KnowledgeSearchResult`

#### `smart-ticket-infra`
目标：承接 Spring AI 配置和外部能力适配。

需要新增或明确这些组件：

- `ai/ChatClientConfig`
- `ai/EmbeddingModelConfig`
- `ai/VectorStoreConfig`
- `ai/ToolConfig`（可选）
- `vector/PgVectorAdapter`（如需要单独适配）
- `redis/`（保留会话与缓存）
- Embedding 模型配置
- pgvector 连接配置

### 3.2 依赖接入清单

需要引入的 Spring AI 方向依赖：

- Chat 模型能力
- Embedding 模型能力
- VectorStore（PGvector）
- 可选 observability

接入原则：

- 固定稳定版本，不追 snapshot
- 与当前 Spring Boot 版本兼容
- 不要同时混用 Spring AI 和 LangChain4j 做主流程

### 3.4 Tool Calling 详细实施清单

总原则：

> Tool 只是业务系统能力的封装层，不是新的业务层。

Tool 必须全部调用现有 `biz` service，而不是重新实现业务逻辑。

#### 3.4.1 `QueryTicketTool`
职责：
- 调 `TicketQueryService`
- 返回当前工单事实数据
- 不接入 RAG

任务：
- [ ] 定义入参模型
- [ ] 定义返回模型
- [ ] 处理上下文 ticketId 推断
- [ ] 增加权限校验失败处理

#### 3.4.2 `CreateTicketTool`
职责：
- 调 `TicketCommandService`
- 支持幂等
- 在正式创建前，可串联“相似案例检查”

任务：
- [ ] 定义创建参数模型
- [ ] 复用现有幂等逻辑
- [ ] 对接多轮澄清后的字段
- [ ] 增加用户确认继续创建的分支

#### 3.4.3 `TransferTicketTool`
职责：
- 调 `TicketAssignService`
- 保留关系权限判断

任务：
- [ ] 支持从上下文取 `activeTicketId`
- [ ] 支持目标处理人解析
- [ ] 保留管理员 / 当前负责人校验

#### 3.4.4 `SearchHistoryTool`
职责：
- 调 `RetrievalService`
- 返回历史案例摘要和分数

任务：
- [ ] 定义检索入参
- [ ] 定义检索结果结构
- [ ] 与 Agent 输出整合
- [ ] 确保输出是“参考”，不是“绝对判断”

### 3.5 Spring AI ChatClient 实施清单

目标：
用 Spring AI 接管模型交互层，而不是手搓模型调用。

要做的事：
- [ ] 在 `infra` 中新增 `ChatClientConfig`
- [ ] 暴露统一 `ChatClient`
- [ ] 配置系统 prompt 注入方式
- [ ] 为四类意图分别设计 prompt 模板
- [ ] 在 `AgentFacade` 或 handler 中调用 `ChatClient`
- [ ] 接入 Tool Calling
- [ ] 保留后续接 Advisor 的位置

建议实施方式：
先不要把所有意图塞到一个超级 prompt 里。第一版建议：

- 一个统一 system prompt
- 每种意图一个 handler / prompt 组合

### 3.6 Spring AI 与当前业务层的边界清单

允许 Spring AI 处理的内容：
- 自然语言理解
- Tool 选择与调用
- 返回结果组织
- 历史案例参考回答
- 多轮澄清话术组织

不允许 Spring AI 直接决定的内容：
- 工单状态是否合法流转
- 用户是否有权转派/关闭
- 幂等是否成立
- 工单是否真正创建成功
- 日志是否写入
- 工单关闭后的知识构建主触发
- 是否直接拒绝创建工单

原则：

> Spring AI 负责“模型交互层”，业务系统负责“业务真规则层”。

1. AgentController / AgentFacade

第三阶段再做更复杂平台能力。

---

## 6. 最终原则

如果你按当前项目结构接入 Spring AI，最重要的判断标准不是：

> 我有没有把 Spring AI 用到很多地方

而是：

> 我有没有用 Spring AI 显著减少 Agent 样板代码，同时保持业务真规则继续由我的 `biz` 模块掌控。

这才是当前项目最稳、最合理、最快的接入方式。
