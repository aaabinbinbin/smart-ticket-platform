# 企业智能工单协同平台开发文档（最新版）

## 1. 项目定位

本项目是一个**以 Java 后端业务系统为主体、以 Agent 能力为增强层**的项目，目标是同时满足以下两类实习岗位的能力展示需求：

- **Java 后端开发实习**：体现业务建模、数据库设计、状态流转、权限控制、缓存、幂等、日志审计、接口设计与工程化能力。
- **Agent 开发实习**：体现意图路由、Tool Calling、RAG、会话上下文、自然语言业务入口、Agent 与业务系统集成能力。

项目名称建议：

- **企业智能工单协同平台**
- **面向服务台场景的智能工单处理系统**

不建议将项目命名为“AI 聊天机器人”“多 Agent Demo”“智能问答助手”，否则会削弱项目在 Java 后端方向上的业务属性。

---

## 2. 为什么选择工单场景

工单场景不是业务复杂度最高的方向，但它是**最容易同时证明 Java 后端工程能力和 Agent 落地能力**的方向之一。

它天然具备以下优点：

1. **业务闭环清晰**：创建、分配、处理、评论、关闭、归档、经验沉淀。
2. **后端能力自然**：用户权限、状态流转、审计日志、幂等、缓存、查询优化都能合理落地。
3. **Agent 能力自然**：自然语言查询工单、创建工单、转派工单、历史案例检索、结果摘要。
4. **适合学生独立实现**：比支付、交易、风控等业务更容易做完整闭环，同时比纯聊天 Demo 更有业务厚度。

需要明确的是：

- 如果只是“聊天框 + 模型 API + 向量检索”，这个项目的业务程度是不够的。
- 如果做成“完整工单系统 + Agent 增强层”，这个项目完全可以稳定写进简历。

## 3. 业务本质

这个项目解决的问题是：

> 企业内部员工遇到问题后，如何将问题正式受理、推进协作处理、留存处理过程，并把解决经验沉淀为后续可复用知识。

系统首先是一个**流程系统**，其次才是一个带 Agent 的系统。

如果企业只靠群聊和私聊处理问题，会很快出现这些问题：

- 消息容易被淹没
- 没有明确责任人
- 不知道处理到哪一步
- 中间处理过程不完整
- 很难统计与审计
- 类似问题反复出现但经验无法沉淀

工单系统的价值在于把：

**“零散沟通”**  
变成  
**“可管理的处理流程”**

## 4. 核心角色与权限位置

### 4.1 业务位置

- **提单人**：这张工单的创建者
- **处理人**：这张工单当前负责人
- **管理员/组长**：负责分配、调整和流程监管的人

### 4.2 系统角色

建议角色：

- `USER`
- `STAFF`
- `ADMIN`

一个用户可以同时拥有多个角色。例如：

- 普通员工：`USER`
- 开发/运维/技术支持：`USER + STAFF`
- 组长/管理员：`USER + STAFF + ADMIN` 或 `ADMIN`

### 4.3 重要原则

提单人和处理人不是固定系统角色，而是用户在某张工单中的**业务位置**。

权限判断时通常要同时看：

1. 用户系统角色
2. 用户与当前工单的关系（创建人 / 当前负责人 / 管理员）
3. 当前工单状态

可以概括为：

> **权限 = 角色能力 + 工单关系 + 当前状态**

## 5. 核心业务规则

### 5.1 工单不是静态记录，而是处理过程

工单管理的不是“问题文本”，而是**问题处理链**。

### 5.2 模糊工单先澄清，不要直接处理

当用户描述过于模糊时，例如：

- 系统有问题
- 接口不行
- 帮忙看看

正确做法不是强行创建高质量工单，而是：

- 先追问关键字段
- 补齐系统、环境、时间、报错、影响范围、截图等信息
- 再进入正式处理或创建

### 5.3 多人参与时必须有一个主负责人

复杂问题可以有很多协作方，但必须始终有一个**当前负责人**来推进工单，而不是多人共同模糊负责。

### 5.4 已解决和已关闭要分开

推荐状态链：

- `PENDING_ASSIGN`
- `PROCESSING`
- `RESOLVED`
- `CLOSED`

其中：

- `RESOLVED`：处理人视角，表示“我这边处理完了，请确认”
- `CLOSED`：提单人或流程视角，表示“问题正式结束并归档”

### 5.5 转派不是普通字段修改，而是责任变更

转派会改变责任链，所以必须：

- 做权限校验
- 明确新负责人
- 写操作日志

## 6. 工单生命周期

一张工单的生命周期可以理解为 6 步：

1. 发现问题
2. 创建工单
3. 分配处理人
4. 处理中协作
5. 问题解决
6. 确认关闭

### 6.1 发现问题
有人发现问题或提出需要处理的请求。

### 6.2 创建工单
把口头问题正式变成系统里的待处理事项。

### 6.3 分配处理人
明确由谁来接手处理。

### 6.4 处理中协作
补充信息、写评论、排查问题、转派、更新状态。

### 6.5 问题解决
处理人给出处理结果，并填写解决方案摘要。

### 6.6 确认关闭
提单人或流程确认问题结束。

> 对用户来说，工单到关闭时结束；  
> 对系统来说，工单在关闭后开始具备知识沉淀价值。

## 7. 三类数据模型

这个项目里，必须区分以下三类数据。

### 7.1 事实数据：现在是什么

例如：

- 当前状态
- 当前负责人
- 工单号
- 优先级
- 创建人

特点：

- 强调当前真相
- 需要精确
- 通常来自 MySQL / Redis

### 7.2 过程数据：怎么变成现在的

例如：

- 评论
- 状态变更记录
- 转派记录
- 附件
- 操作日志

特点：

- 强调过程和轨迹
- 要求可追溯
- 服务于协作和审计

### 7.3 知识数据：以后还能怎么用

例如：

- 已关闭工单的处理经验
- 常见故障的处理方式
- 可复用的历史案例

特点：

- 服务于经验复用
- 适合知识检索和 RAG
- 不要求事务级实时强一致

一句话记忆：

- **事实数据：现在是什么**
- **过程数据：怎么变成现在的**
- **知识数据：以后还能怎么用**

## 8. Agent 的职责边界

### 8.1 Agent 适合做什么

Agent 适合做：

- 意图识别
- 澄清追问
- 结构化整理
- Tool 调用编排
- 历史案例检索触发
- 结果摘要

也就是说：

> Agent 负责“听懂、问清、整理、检索、表达”

### 8.2 Agent 不适合做什么

Agent 不应承担：

- 当前事实真相来源
- 越权执行
- 信息不足时强行写操作
- 严格规则判断
- 事务一致性控制

也就是说：

> 后端负责“校验、执行、留痕、一致性”  
> Agent 负责“理解、辅助、组织输出”

### 8.3 RAG 不是全局默认前置步骤

不是每次自然语言请求都要先走 RAG。

应先判断请求类型：

- **事实查询类**：查数据库 / Redis / Tool
- **业务执行类**：参数提取 + 业务 Service + Tool
- **知识检索类**：走 RAG

第一版项目中，RAG 最自然的主入口是：

- `SEARCH_HISTORY`

但在 `CREATE_TICKET` 链路中，可以加入一个**创建前相似案例检查**步骤，用于减少重复工单。

## 9. MVP 设计原则

MVP 不是“最简陋版本”，而是：

> **最小但完整、能跑通核心业务闭环的版本**

目标是做到：

- 业务闭环完整
- 后端能力自然
- Agent 有亮点
- 面试可解释
- 简历可稳定书写

### 9.1 MVP 必做能力

#### 用户与权限
- 登录认证
- 基础角色权限控制
- 当前用户只能访问自己可见范围内的工单

#### 工单核心流程
- 创建工单
- 查询工单详情
- 查询工单列表
- 分配 / 转派工单
- 更新状态
- 添加评论
- 关闭工单

#### 工单辅助能力
- 分类
- 优先级
- 附件上传（MVP 先存 URL）
- 操作日志

#### Agent 能力
- 自然语言查工单
- 自然语言创建工单
- 自然语言转派工单
- 历史工单检索

#### RAG 能力
- 已关闭工单生成知识文本
- 向量化入库
- 相似历史工单召回
- 创建前相似案例检查
- 检索结果供 Agent 总结输出

#### 工程化能力
- Redis 工单详情缓存
- 创建工单幂等防重
- Swagger/OpenAPI
- Docker Compose
- 单元测试

### 9.2 第一版暂不做

- 复杂审批流
- 子工单/子任务系统
- SLA 自动升级
- 多渠道接入（飞书/企业微信/邮件）
- MCP
- 复杂多 Agent 并行
- 多模态
- LLM Wiki
- Elasticsearch 混合检索
- 高级报表
- 重 DDD 架构

## 10. 四条 MVP 主业务链路

### 10.1 创建工单链路

目标：把问题正式受理进系统。

关键业务点：

1. 用户发起请求
2. 信息不足时先澄清
3. 可选地做“创建前相似案例检查”
4. 创建正式工单
5. 进入初始状态 `PENDING_ASSIGN`
6. 记录创建日志
7. 返回工单号和当前状态

### 10.2 查询工单链路

目标：返回当前业务真相。

关键业务点：

1. 识别查询对象
2. 查当前状态、当前负责人、最近处理进展
3. 做权限判断
4. 可由 Agent 对结果做自然语言摘要

注意：  
这条链路主要依赖数据库 / 缓存，不应主要依赖 RAG。

### 10.3 转派工单链路

目标：完成责任人变更。

关键业务点：

1. 识别工单对象和目标处理人
2. 做严格权限校验
3. 更新负责人
4. 写转派日志
5. 返回结果

### 10.4 历史案例检索链路

目标：复用历史处理经验。

关键业务点：

1. 识别这是知识型问题
2. 做查询改写
3. 检索历史工单知识
4. 返回相似案例和参考方案
5. 以“参考”方式呈现，而不是给出绝对结论

## 11. 总体技术架构

### 11.1 架构形态

本项目采用：

> **模块化单体**

而不是微服务。

原因：

- 对学生项目更稳
- 第一版重点是业务闭环和 Agent 集成，不是分布式治理
- 模块化单体已经足够体现工程能力

### 11.2 系统总分层

系统可以理解为两大层：

#### 第一层：业务后端主系统
负责：

- 认证授权
- 工单生命周期
- 评论与附件
- 操作日志
- 缓存、幂等、审计
- 关系型数据存储

#### 第二层：智能增强层
负责：

- 意图路由
- Tool 调用
- 历史案例检索
- 会话上下文管理
- 自然语言结果组织

## 12. 推荐技术栈

- **语言与框架**：Java 17、Spring Boot 3.x
- **认证授权**：Spring Security + JWT
- **数据库**：MySQL（主业务库）
- **缓存**：Redis
- **向量数据库**：PostgreSQL + pgvector
- **Agent 框架**：优先建议 Spring AI
- **文档**：Swagger / OpenAPI
- **测试**：JUnit 5、Mockito
- **部署**：Docker、Docker Compose

### 12.1 为什么向量库优先选择 pgvector

当前项目第一版建议使用：

- **Redis**：缓存工单详情、会话上下文、幂等键
- **pgvector**：存储历史工单知识向量、做相似案例检索

原因：

- Redis 虽然也能做向量搜索，但对你这个项目第一版没有必要承担向量主存职责
- pgvector 更适合“业务数据 + 知识数据”一起管理的业务系统型项目
- 更容易向面试官解释：Redis 负责缓存，pgvector 负责历史经验检索

## 13. 目录与模块设计

推荐 Maven 多模块结构如下：

```text
smart-ticket-platform
├─ smart-ticket-app
├─ smart-ticket-common

├─ smart-ticket-domain
├─ smart-ticket-biz
├─ smart-ticket-agent
├─ smart-ticket-rag
├─ smart-ticket-infra
└─ smart-ticket-api
```

### 13.1 模块职责

#### `smart-ticket-app`
负责：

- Spring Boot 启动类
- 全局配置装配
- 全局异常处理
- Swagger 配置
- 统一启动入口

#### `smart-ticket-common`
负责：

- 通用常量
- 错误码
- 通用异常
- 统一返回体
- 工具类
- 基础枚举父类

#### `smart-ticket-auth`
负责：

- 登录认证
- JWT
- Spring Security 配置
- 当前用户上下文
- 基础 RBAC

注意：  
`auth` 负责“你是谁”；  
“你能不能操作这张工单”应由 `biz` 结合业务关系判断。

#### `smart-ticket-domain`
负责：

- Entity / PO
- 枚举
- Mapper / Repository 接口
- 核心数据对象定义

建议放：

- `SysUser`
- `SysRole`
- `SysUserRole`
- `Ticket`
- `TicketComment`
- `TicketOperationLog`
- `TicketAttachment`
- `TicketKnowledge`
- `TicketKnowledgeEmbedding`

#### `smart-ticket-biz`
负责：

- 工单核心业务规则
- 状态流转
- 分配与转派
- 评论与关闭
- 幂等逻辑
- 权限关系判断
- 操作日志写入
- 知识构建触发

建议服务：

- `TicketQueryService`
- `TicketCommandService`
- `TicketAssignService`
- `TicketCommentService`
- `TicketPermissionService`
- `TicketKnowledgeService`

#### `smart-ticket-agent`
负责：

- Agent 对话入口
- 意图识别
- Prompt 模板
- Tool 封装
- 会话上下文
- 路由和结果摘要

注意：  
`agent` 不能直接改数据库，也不能绕过 `biz`。

#### `smart-ticket-rag`
负责：

- 知识切片
- embedding
- query rewrite
- 向量检索
- 历史案例召回

建议服务：

- `EmbeddingService`
- `RetrievalService`
- `QueryRewriteService`
- `TextSplitService`

#### `smart-ticket-infra`
负责：

- Redis 封装
- 向量库适配
- 文件存储
- LLM Client
- Embedding Client
- 第三方 SDK 封装
- 外部基础设施配置

#### `smart-ticket-api`
负责：

- Controller
- DTO / VO
- 参数校验
- Assembler / Converter
- HTTP 接口层

### 13.2 推荐依赖关系

建议遵守以下依赖方向：

```text
api -> biz
api -> agent

agent -> biz
agent -> rag

biz -> domain
biz -> infra
biz -> rag（仅在需要触发知识向量化时做受控依赖，尽量通过接口或异步任务隔离）

rag -> domain
rag -> infra

auth -> common
biz -> common
agent -> common
rag -> common
infra -> common

app -> api / auth / biz / agent / rag / infra / domain / common
```

重要原则：

- `api` 不直接操作 `domain`
- `agent` 不绕过 `biz`
- `rag` 不参与主事务写逻辑
- `infra` 不反向依赖业务模块

## 14. 数据库表结构设计

MVP 阶段至少实现以下 9 张核心表。

### 14.1 `sys_user`

```sql
CREATE TABLE sys_user (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(64) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    real_name VARCHAR(64) NOT NULL,
    email VARCHAR(128),
    status TINYINT NOT NULL DEFAULT 1 COMMENT '1-启用 0-禁用',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

### 14.2 `sys_role`

```sql
CREATE TABLE sys_role (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    role_code VARCHAR(64) NOT NULL UNIQUE,
    role_name VARCHAR(64) NOT NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

建议角色：

- `USER`
- `STAFF`
- `ADMIN`

### 14.3 `sys_user_role`

```sql
CREATE TABLE sys_user_role (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    role_id BIGINT NOT NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_user_role (user_id, role_id)
);
```

### 14.4 `ticket`

```sql
CREATE TABLE ticket (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    ticket_no VARCHAR(32) NOT NULL UNIQUE,
    title VARCHAR(200) NOT NULL,
    description TEXT NOT NULL,
    category VARCHAR(64) NOT NULL COMMENT 'ACCOUNT/SYSTEM/ENVIRONMENT/OTHER',
    priority VARCHAR(32) NOT NULL COMMENT 'LOW/MEDIUM/HIGH/URGENT',
    status VARCHAR(32) NOT NULL COMMENT 'PENDING_ASSIGN/PROCESSING/RESOLVED/CLOSED',
    creator_id BIGINT NOT NULL,
    assignee_id BIGINT DEFAULT NULL,
    solution_summary TEXT,
    source VARCHAR(32) NOT NULL DEFAULT 'MANUAL' COMMENT 'MANUAL/AGENT',
    idempotency_key VARCHAR(128) DEFAULT NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_creator_id (creator_id),
    INDEX idx_assignee_id (assignee_id),
    INDEX idx_status (status),
    INDEX idx_category (category),
    INDEX idx_created_at (created_at)
);
```

### 14.5 `ticket_comment`

```sql
CREATE TABLE ticket_comment (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    ticket_id BIGINT NOT NULL,
    commenter_id BIGINT NOT NULL,
    comment_type VARCHAR(32) NOT NULL COMMENT 'USER_REPLY/PROCESS_LOG/SOLUTION',
    content TEXT NOT NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_ticket_id (ticket_id),
    INDEX idx_commenter_id (commenter_id)
);
```

### 14.6 `ticket_operation_log`

```sql
CREATE TABLE ticket_operation_log (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    ticket_id BIGINT NOT NULL,
    operator_id BIGINT NOT NULL,
    operation_type VARCHAR(64) NOT NULL COMMENT 'CREATE/ASSIGN/TRANSFER/UPDATE_STATUS/COMMENT/CLOSE',
    operation_desc VARCHAR(500) NOT NULL,
    before_value TEXT,
    after_value TEXT,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_ticket_id (ticket_id),
    INDEX idx_operator_id (operator_id)
);
```

### 14.7 `ticket_attachment`

```sql
CREATE TABLE ticket_attachment (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    ticket_id BIGINT NOT NULL,
    file_name VARCHAR(255) NOT NULL,
    file_url VARCHAR(500) NOT NULL,
    file_type VARCHAR(64),
    file_size BIGINT,
    uploader_id BIGINT NOT NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_ticket_id (ticket_id)
);
```
