# Codex Task Plan：按当前项目结构接入 Spring AI，并更新完整版智能工单平台路线图

## 1. 文档用途

这份文档专门用于：

- 把当前项目按现有结构接入 Spring AI
- 把 Spring AI 接入任务拆给 Codex 分阶段执行
- 同时保持完整版智能工单平台的 P0 / P1 / P2 路线清晰

本版新增重点：

- Spring AI 接入策略
- Spring AI 在 `agent / rag / infra` 的落位
- Agent / Tool / RAG 接入的分阶段任务
- 明确哪些代码必须保留在 `biz` 模块，不允许被 Spring AI 接管

---

## 2. 总原则

### 2.1 不重写业务主系统

保留以下模块现有定位：

- `smart-ticket-biz`：业务真规则
- `smart-ticket-domain`：实体与仓储接口
- `smart-ticket-auth`：认证授权
- `smart-ticket-api`：HTTP 接口层
- `smart-ticket-rag`：知识向量化与检索
- `smart-ticket-infra`：外部依赖适配

### 2.2 Spring AI 接管范围

Spring AI 只接管：

- ChatClient
- Tool Calling
- EmbeddingModel
- VectorStore
- Advisor
- 可选 observability

### 2.3 不允许 Spring AI 接管的范围

这些必须继续由项目自己的 `biz` 模块负责：

- 工单状态流转
- 权限判断
- 幂等
- 操作日志
- 工单关闭主流程
- TicketKnowledgeService 的业务准入判断
- 创建工单最终是否成功
- 转派是否合法

---

## 3. 模块落位说明

### `smart-ticket-agent`
负责：
- AgentController
- AgentFacade
- IntentRouter
- Tool 封装
- Session 上下文
- per-intent handler
- 基于 Spring AI 的 ChatClient 调用

### `smart-ticket-rag`
负责：
- EmbeddingService
- RetrievalService
- QueryRewriteService
- TextSplitService
- 基于 Spring AI 的 EmbeddingModel / VectorStore 使用

### `smart-ticket-infra`
负责：
- ChatClientConfig
- EmbeddingModelConfig
- VectorStoreConfig
- Redis 配置
- pgvector 连接
- Embedding 模型客户端配置

---

## 4. 当前项目接入 Spring AI 的总施工顺序

建议严格按这个顺序来：

1. 基线清理与文档同步
2. 引入 Spring AI 依赖与配置
3. `smart-ticket-agent` 接入 ChatClient
4. Tool Calling 打通
5. Session 与上下文接入
6. `smart-ticket-rag` 接入 EmbeddingModel
7. `smart-ticket-rag` 接入 VectorStore(PGvector)
8. RetrievalService 打通
9. 创建前相似案例检查接入
10. 补测试与 demo
11. 再继续推进 P1 / P2

### 5.4 P0-3：Tool Calling 接入 Spring AI

发给 Codex 的任务：

```text
请基于 Spring AI Tool Calling 能力，接入当前项目的四个核心 Tool。

要求：
1. 在 smart-ticket-agent 中实现：
   - QueryTicketTool
   - CreateTicketTool
   - TransferTicketTool
   - SearchHistoryTool
2. 每个 Tool 必须调用现有 biz service，不允许复制业务逻辑
3. Tool 的职责：
   - queryTicket：只查当前事实，不接 RAG
   - createTicket：调用 TicketCommandService，复用幂等逻辑
   - transferTicket：调用 TicketAssignService，保留权限判断
   - searchHistory：调用 rag 检索链路
4. 在 AgentFacade 中接入 Tool Calling
5. 在 docs/tool-design.md 中说明：
   - Tool 输入输出
   - Tool 风险边界
   - 为什么 Tool 不能替代 biz

限制：
- 任何写操作都必须通过 biz 模块
- 不允许直接从 agent 操作 repository
```

## 6. P1 / P2 路线保留并延续

以下阶段原则不变，但后续尽量继续沿用 Spring AI 作为 Agent 层主框架。

### 6.1 P1：业务复杂度与 Agent 协同升级
继续推进：
- 队列 / 组模型
- SLA
- 多类型工单
- 简单审批流
- 智能分诊
- 多轮澄清
- 阶段摘要
- 权限细化
- 基础评测

要求：
- Spring AI 继续用于 Agent 交互层
- 业务真规则继续留在 biz
- RAG 检索与创建前分流继续复用 rag 模块

### 6.2 P2：平台级升级
继续推进：
- 子工单 / 子任务
- 相似工单聚类
- 知识质量治理
- 主动治理 Agent
- 可观测性
- 多租户 / 多团队
- 运营指标

要求：
- 不破坏当前模块化单体结构
- 继续保持 agent / rag / biz / infra 边界清晰

## 8. 最终目标

按这份文档推进后，你应该得到一个：

- 保留现有后端业务骨架
- Agent 层由 Spring AI 明显提速
- Tool Calling、RAG、创建前分流真正打通
- 后续仍能自然推进 P1 / P2

的完整版智能工单平台路线。

最关键的设计准则只有一句话：

> 让 Spring AI 简化 Agent 开发速度，但绝不让它侵入和替代你的业务真规则层。
