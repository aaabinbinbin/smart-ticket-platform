# 智能工单平台：代码审查修复与后续优化执行清单

## 1. 文档目的

这份文档将两类内容合并整理：

1. **代码审查修复清单**
2. **业务与 Agent 后续优化清单**

目标不是简单罗列问题，而是：

> **给当前项目安排一个清晰、可执行、有优先级的推进顺序**

适用场景：

- 你自己后续开发排期
- 拆任务给 Codex
- 更新 README / current-status / 开发文档
- 面试前判断“当前项目还有哪些关键短板”

---

## 2. 当前项目总体判断

当前项目已经具备：

- 模块化单体骨架
- 工单核心业务闭环
- Spring AI 驱动的 Agent 入口
- Tool Calling 基础闭环
- 工单关闭后的知识构建
- embedding / 历史案例检索基础链路
- P1 初步能力：工单组 / 队列 / SLA / 自动分派最小闭环

因此：

> **当前项目不需要推倒重做，而是需要“先修关键工程问题，再继续向完整版平台扩展”。**

---

## 3. 执行顺序总览

后续任务建议按下面 4 个阶段推进：

### 阶段 A：先修关键可信度问题
目标：让仓库“更可信、更稳、更像一个完成度高的项目”。

### 阶段 B：补强当前主链路
目标：让现有业务、Agent、RAG 三条链更扎实，不容易在演示和面试时翻车。

### 阶段 C：推进高价值业务复杂度
目标：让平台更像真实企业服务台，而不是只有工单 MVP。

### 阶段 D：推进平台级能力
目标：进一步拉高项目上限，向完整平台演进。

---

# 4. 阶段 A：必须先修的关键问题（Must Fix）

这一阶段的任务，不优先做会直接影响：

- 仓库可信度
- 简历可信度
- 面试时的可解释性
- 后续继续迭代的稳定性

---

## A1. 统一 README、ticket-api、current-status 与代码现状

### 问题
当前仓库存在明显的文档状态漂移：

- README 中仍有“当前仅完成工程初始化、暂未实现具体业务逻辑”之类旧描述
- `docs/ticket-api.md` 中部分内容仍把 Agent / RAG 写成“暂不包含”
- `docs/current-status.md` 已经写到 P0 / P1 不少能力

这会让审阅者无法判断仓库到底完成到了哪一步。

### 要做的事
- [ ] 重写 README 的“当前进展”部分
- [ ] 更新 `docs/ticket-api.md`
- [ ] 校正 `docs/current-status.md`
- [ ] 增加一个“已完成 / 未完成 / 下一步”总表
- [ ] 确保三个文档表达一致

### 交付物
- 新版 README
- 新版 `docs/ticket-api.md`
- 新版 `docs/current-status.md`

### 为什么优先级最高
因为这项成本低，但收益非常高。  
如果文档都不一致，别人会默认你项目还没真正落地。

---

## A2. 统一类注释与当前实现状态

### 问题
当前有些类的注释已经落后于真实实现状态，比如：

- SLA 服务注释仍强调不做自动化扫描/升级
- Assignment rule 服务注释仍偏 preview-only
- 但 `current-status.md` 已写到更多进展

### 要做的事
- [ ] 检查 `smart-ticket-biz` 中核心 service 的类注释
- [ ] 检查 `smart-ticket-agent` 中 facade / router / handler 的注释
- [ ] 检查 `smart-ticket-rag` 中 embedding / retrieval 相关注释
- [ ] 删除或改写已经失效的类级说明
- [ ] 为关键方法补准确注释

### 交付物

### 为什么重要
类注释就是“给未来的你”和“给面试官的代码自说明”。  
如果注释失真，代码可信度会下降。

## A4. 补 Tool 边界测试

### 问题

- queryTicket 错走 RAG
- createTicket 绕过幂等
- transferTicket 绕过权限
- searchHistory 输出不稳定

### 要做的事
- [ ] 为四个 Tool 分别补测试
- [ ] 明确每个 Tool 的：
  - [ ] 输入契约
  - [ ] 权限边界
  - [ ] 错误响应
  - [ ] 与业务 service 的依赖关系
- [ ] 确保 Tool 不直接访问 repository

### 交付物
- Tool 层测试
- Tool 边界文档（可选）

---

# 5. 阶段 B：补强主链路（Should Fix）

这一阶段的目标是：

> **让现有业务主链、Agent 主链、RAG 主链从“能跑”升级成“更扎实、更可放大”。**

---

## B1. 拆分 TicketService，收紧业务服务边界

### 问题
`TicketService` 当前已经承担太多职责：

- 创建
- 分配
- 转派
- 状态流转
- 评论
- 关闭
- 权限判断
- 缓存
- 日志
- afterCommit 知识触发

继续往后扩展会越来越难维护。

### 要做的事
- [ ] 识别 TicketService 中的职责块
- [ ] 按职责拆分为：
  - [ ] `TicketCommandService`
  - [ ] `TicketQueryService`
  - [ ] `TicketWorkflowService`
  - [ ] `TicketCommentService`
  - [ ] `TicketQueueBindingService`
- [ ] 逐步迁移 controller / tool 依赖
- [ ] 保留对外兼容接口（如需要）

### 交付物
- 更清晰的业务服务边界

### 预期收益
- 更容易加审批流 / 子任务 / 更复杂 SLA
- 更容易做单测
- 面试时更好讲

---

## B2. 升级 IntentRouter：从纯关键词路由到“规则 + 低置信度澄清”

### 问题
当前 IntentRouter 偏关键词驱动，容易出现：

- 混合意图误判
- 默认兜底进 QUERY_TICKET
- 复杂表达无法正确路由

### 要做的事
- [ ] 增加低置信度判定
- [ ] 对低置信度请求先澄清，不直接硬路由
- [ ] 引入简单优先级规则（例如 transfer > query）
- [ ] 区分“历史案例问题”与“当前工单问题”
- [ ] 保留后续模型辅助分类接口

### 交付物
- 更稳的 IntentRouter
- 更自然的错误/澄清分支

---

## B3. 补全多轮澄清创建工单

### 要做的事

- [ ] 按工单类型设计澄清问题
- [ ] 让会话上下文保存“待创建工单草稿”
- [ ] 澄清完成后再调用 CreateTicketTool
- [ ] 对超时中断和取消创建做处理

### 交付物
- 多轮澄清创建工单链路

### 预期收益
- Agent 体验明显更自然
- 业务数据质量更高

# 10. 最终结论

当前项目已经足够支撑：

> **智能工单平台初版 / MVP**

后续最优策略不是继续无序加功能，而是：

1. 先修文档、测试、边界和主链稳定性问题
2. 再补强当前 Agent + RAG + 业务链
3. 最后推进复杂业务和平台化能力

这样你的项目会越来越像一个：

- 能讲清楚
- 能跑通
- 能写进简历
- 也能继续放大

的真实平台项目。
