# Spring AI Tool Design Change 3

更新时间：2026-04-19

## 1. 本次变更目标

本次基于 Spring AI Tool Calling 接入当前四个核心 Tool：

- `queryTicket`
- `createTicket`
- `transferTicket`
- `searchHistory`

这些 Tool 都位于 `smart-ticket-agent`，并继续复用现有 `AgentTool` 抽象、`AgentExecutionGuard` 和 `biz/rag` 服务链路。

## 2. Tool 输入输出

### `queryTicket`

职责：

- 查询当前工单事实数据。
- 不接入 RAG。

输入：

- `ticketId`：可选。存在时查询指定工单详情；为空时查询当前用户可见工单列表。

输出：

- 指定 `ticketId` 时返回 `TicketDetailDTO`。
- 未指定 `ticketId` 时返回 `PageResult<Ticket>`。

调用边界：

```text
queryTicket -> TicketService.getDetail / TicketService.pageTickets
```

### `createTicket`

职责：

- 创建新工单。
- 创建前可返回相似历史案例作为参考。
- 最终创建必须走 biz。

输入：

- `title`：必填，工单标题。
- `description`：必填，问题描述。
- `category`：可选，默认 `OTHER`。
- `priority`：可选，默认 `MEDIUM`。
- `idempotencyKey`：可选，创建幂等键。

输出：

- 创建后的 `Ticket`。
- `similarCases`：创建前相似案例检索结果，仅作参考。

调用边界：

```text
createTicket
  -> RetrievalService.checkSimilarCasesBeforeCreate
  -> TicketService.createTicket
```

当前项目尚未拆出 `TicketCommandService`，所以本阶段调用现有 `TicketService.createTicket`。幂等逻辑仍由 biz 内部 `TicketIdempotencyService` 承接。

### `transferTicket`

职责：

- 将处理中的工单转派给目标处理人。
- 保留权限、状态和目标处理人合法性判断。

输入：

- `ticketId`：可选；为空时可从会话上下文中的当前工单推断。
- `assigneeId`：必填，目标处理人用户 ID。

输出：

- 转派后的 `Ticket`。
- 如果缺少参数或需要确认，返回 `NEED_MORE_INFO`。

调用边界：

```text
transferTicket -> AgentExecutionGuard -> TicketService.transferTicket
```

当前项目尚未拆出 `TicketAssignService`，所以本阶段调用现有 `TicketService.transferTicket`。权限和状态判断仍在 biz 内部。

### `searchHistory`

职责：

- 检索历史工单经验。
- 只作为参考，不裁决当前工单事实。

输入：

- `queryText`：可选；为空时使用用户原始消息。

输出：

- `RetrievalResult`
- 当前会话近期消息 `recentMessages`

调用边界：

```text
searchHistory -> RetrievalService.retrieve
```

## 3. Spring AI 接入方式

四个 Tool 在原有 `AgentTool.execute()` 基础上新增 Spring AI `@Tool` 方法。

Spring AI 调用链路：

```text
AgentFacade
  -> IntentRouter 先确定四类意图之一
  -> 只向 ChatClient 暴露该意图对应的一个 Tool
  -> Spring AI 选择并填充 Tool 参数
  -> SpringAiToolSupport
  -> AgentExecutionGuard
  -> AgentTool.execute()
  -> biz / rag service
```

这样做的原因：

- 避免把四类意图全部塞进一个超级 prompt。
- 避免模型越权调用本轮不该调用的 Tool。
- 避免 Spring AI 原生 Tool 直接绕过项目已有 Guard。

如果 Spring AI 未启用、模型未调用工具或调用失败，`AgentFacade` 会回退到现有 `TicketAgentOrchestrator`。

## 4. Tool 风险边界

### 只读 Tool

- `queryTicket`
- `searchHistory`

风险等级：

- `READ_ONLY`

约束：

- `queryTicket` 只查询当前事实，不走 RAG。
- `searchHistory` 只查历史经验，不替代当前事实。

### 写操作 Tool

- `createTicket`
- `transferTicket`

风险等级：

- `createTicket`：`LOW_RISK_WRITE`
- `transferTicket`：`HIGH_RISK_WRITE`

约束：

- 写操作必须通过 biz 服务。
- 不允许 agent 直接操作 repository。
- `transferTicket` 必须经过 `AgentExecutionGuard` 的确认与参数检查。
- 最终权限、状态和并发判断仍由 biz 负责。

## 5. 为什么 Tool 不能替代 biz

Tool 是模型与系统能力之间的受控适配层，不是业务规则层。

Tool 不能替代 biz，原因如下：

- 模型输出不稳定，不能作为权限、状态机、幂等和审计的唯一依据。
- Tool 参数来自自然语言抽取，必须被代码校验。
- 工单写操作需要事务、并发控制、缓存失效和操作日志。
- 权限判断必须基于当前登录用户、角色、工单关系和工单状态。
- RAG 命中只代表历史参考，不代表当前请求应被接受或拒绝。

因此边界必须保持为：

```text
Spring AI Tool Calling = 选择工具 + 填充参数
Agent Guard = 执行前安全边界
biz = 业务真规则与写入事实
rag = 历史知识检索参考
infra = 外部模型、Redis、PGvector 等能力适配
```

## 6. 当前限制

- 本阶段没有引入 MCP。
- 本阶段没有做多 Agent 编排。
- 本阶段没有重写工单业务逻辑。
- 当前项目仍使用 `TicketService` 作为 biz 主服务；后续如拆分 `TicketCommandService` / `TicketAssignService`，Tool 只需要切换调用入口，不应复制业务逻辑。
