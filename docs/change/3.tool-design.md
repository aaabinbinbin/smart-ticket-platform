# Tool Design Change 3

更新时间：2026-04-20

## 总体原则

Tool 是业务能力的受控入口，不是新的业务层。

- Tool 可以被 Spring AI Tool Calling 触发。
- Tool 也可以被 `AgentFacade` 的确定性兜底链路触发。
- Tool 必须调用现有 `biz` 或 `rag` service。
- Tool 不允许直接访问 repository。
- 任何写操作都必须经过 `biz`。

## Tool 清单

### queryTicket

- 意图：`QUERY_TICKET`
- 输入：`ticketId`
- 输出：当前工单事实数据。
- 调用：`TicketQueryService`
- 边界：只查当前事实，不接 RAG。

### createTicket

- 意图：`CREATE_TICKET`
- 输入：`title`、`description`、`category`、`priority`、`idempotencyKey`
- 输出：创建后的工单信息。
- 调用：`TicketCommandService`
- 边界：复用 biz 幂等逻辑，不在 agent 内生成业务状态。

### transferTicket

- 意图：`TRANSFER_TICKET`
- 输入：`ticketId`、`assigneeId`
- 输出：转派结果。
- 调用：`TicketAssignService`
- 边界：保留权限判断；高风险写操作必须经过 Guard。

### searchHistory

- 意图：`SEARCH_HISTORY`
- 输入：`queryText`、`topK`
- 输出：历史知识命中列表。
- 调用：`RetrievalService`
- 边界：只返回参考案例，不代表当前工单事实。

## 风险边界

- `QUERY_TICKET`：读操作，风险低。
- `SEARCH_HISTORY`：读操作，风险低，但结果只能作为参考。
- `CREATE_TICKET`：写操作，必须走 `TicketCommandService`。
- `TRANSFER_TICKET`：高风险写操作，必须保留确认、权限和必填参数校验。

## 为什么 Tool 不能替代 biz

`biz` 承担事实规则：

- 工单状态机。
- 权限判断。
- 幂等。
- 业务异常码。
- 操作日志。
- 知识生产条件。

Tool 只负责把 Agent 输入转换成一次受控业务调用。模型输出不可信，不能直接成为业务事实。
