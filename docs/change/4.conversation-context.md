# Conversation Context Change 4

更新时间：2026-04-20

## 1. 本次变更目标

本次在 `smart-ticket-agent` 中收敛会话上下文管理，并与 `AgentFacade` 和 Spring AI Tool Calling 链路集成。

第一版只做短上下文：

- 不做长期记忆
- 不做复杂实体图谱
- 不做跨会话用户画像

## 2. Redis Key

Agent 会话上下文存储在 Redis：

```text
agent:session:{sessionId}
```

key 由 `RedisKeys.agentSession(sessionId)` 统一生成。

当前 TTL 沿用 `AgentSessionCacheService`：

```text
2 hours
```

## 3. 上下文字段

会话上下文对象为：

```text
AgentSessionContext
```

第一版字段：

- `activeTicketId`：最近一次查询、创建或操作涉及的工单 ID。
- `activeAssigneeId`：最近一次确认或操作涉及的处理人 ID。
- `lastIntent`：最近一次识别出的意图编码。
- `recentMessages`：最近若干轮用户消息摘要。
- `pendingAction`：已有编排链路中的待补参或待确认动作。

本次要求的四个字段均已包含：

```text
activeTicketId
activeAssigneeId
lastIntent
recentMessages
```

## 4. 新增 AgentSessionService

新增：

```text
smart-ticket-agent/src/main/java/com/smartticket/agent/service/AgentSessionService.java
```

职责：

- 调用前加载上下文：`load(sessionId)`
- 调用后保存上下文：`save(sessionId, context)`
- Tool 执行后更新上下文：`updateAfterTool(...)`
- 最小指代消解：`resolveReferences(...)`
- 清理会话：`clear(sessionId)`

底层仍委托：

```text
AgentSessionCacheService -> RedisJsonClient -> Redis
```

这样保留原 Redis 读写实现，同时把 AgentFacade 和 Tool Calling 需要的上下文用例集中到一个服务中。

## 5. AgentFacade 集成

`AgentFacade` 当前流程：

```text
chat(currentUser, sessionId, message)
  -> AgentSessionService.load(sessionId)
  -> 判断 Spring AI ChatClient 是否可用
  -> 可用时尝试 Spring AI Tool Calling
      -> IntentRouter 路由
      -> toolContext 携带 CurrentUser / AgentSessionContext / 原始消息 / 路由结果
      -> Tool 执行
      -> AgentSessionService.updateAfterTool(...)
  -> 不可用或失败时回退 TicketAgentOrchestrator
```

说明：

- Spring AI Tool Calling 分支由 `AgentFacade` 显式加载并更新上下文。
- 回退分支仍复用既有 `TicketAgentOrchestrator`，该编排器内部已有 Redis 会话加载和保存逻辑。
- 这样避免在回退分支重复写入 `recentMessages`。

## 6. 最小指代消解

实现位置：

```text
AgentContextReferenceResolver
```

接入位置：

```text
SpringAiToolSupport
  -> AgentSessionService.resolveReferences(...)
  -> AgentExecutionGuard
```

也就是说，模型抽取出的 Tool 参数会先经过短上下文补齐，再进入 Guard 校验。

当前支持的工单指代：

- `刚才那个工单`
- `刚刚那个工单`
- `它`
- `这个`
- `这个问题`
- `该问题`
- `这个工单`
- `该工单`

当前支持的处理人指代：

- `他`
- `她`
- `这个处理人`
- `刚才那个处理人`
- `刚刚那个处理人`

示例：

```text
用户：查一下 100 号工单
上下文：activeTicketId=100

用户：把刚才那个工单转给 3
指代消解：ticketId=100, assigneeId=3
```

```text
用户：这个问题现在怎么样了
上下文：activeTicketId=100
指代消解：ticketId=100
```

## 7. 边界

会话上下文只保存对话指针，不保存工单事实副本。

工单事实仍以：

```text
MySQL + biz 查询结果
```

为准。

上下文不能替代：

- 权限判断
- 状态判断
- 工单是否存在
- 目标处理人是否合法
- 工单创建或转派是否成功

这些仍由 `biz` 层负责。

## 8. 当前限制

- 第一版只维护 2 小时短上下文。
- `recentMessages` 只保留有限条数，避免无限增长。
- 指代消解只做关键词级别，不做复杂 NLP。
- 不做长期记忆。
- 不做跨 session 的上下文合并。
