# Agent Chat API Change 2

更新时间：2026-04-19

## 1. 本次变更目标

本次把 Agent HTTP 入口落到 `smart-ticket-api`，并通过 `smart-ticket-agent` 的 `AgentFacade` 接入现有 Agent 编排能力和 Spring AI `ChatClient` 可用性检测。

本阶段不做：

- 多 Agent 编排
- MCP
- 超级 prompt
- 工单业务逻辑重写

## 2. 接口说明

### POST `/api/agent/chat`

用途：

- 自然语言查询工单
- 自然语言创建工单
- 自然语言转派工单
- 检索历史经验

第一版只支持四类意图：

- `QUERY_TICKET`
- `CREATE_TICKET`
- `TRANSFER_TICKET`
- `SEARCH_HISTORY`

### 请求体

```json
{
  "sessionId": "demo-session-001",
  "message": "帮我查一下 100 号工单进度"
}
```

字段说明：

- `sessionId`：必填，最大 128 字符，用于维护 Agent 短期会话上下文。
- `message`：必填，最大 2000 字符，用户自然语言消息。

### 响应体

接口统一包在 `ApiResponse` 中，`data` 示例：

```json
{
  "sessionId": "demo-session-001",
  "intent": "QUERY_TICKET",
  "reply": "已查询到工单信息。",
  "route": {
    "intent": "QUERY_TICKET",
    "confidence": 0.82,
    "reason": "命中查询关键字"
  },
  "context": {
    "activeTicketId": 100,
    "activeAssigneeId": null,
    "lastIntent": "QUERY_TICKET",
    "recentMessages": [
      "帮我查一下 100 号工单进度"
    ],
    "pendingAction": null
  },
  "result": {},
  "springAiChatReady": false
}
```

字段说明：

- `sessionId`：当前会话 ID。
- `intent`：本轮识别出的意图。
- `reply`：面向用户展示的回复。
- `route`：意图路由细节。
- `context`：本轮执行后的短期上下文。
- `result`：Tool 或编排器返回的结构化结果。
- `springAiChatReady`：当前请求处理时 Spring AI `ChatClient` 是否可用。

## 3. 模块落位

### `smart-ticket-api`

新增：

- `AgentController`
- `AgentChatRequest`
- `AgentChatResponse`

职责：

- 暴露 `POST /api/agent/chat`
- 做 HTTP 参数校验
- 从 `Authentication` 转换 `CurrentUser`
- 调用 `AgentFacade`
- 组装 HTTP 响应

### `smart-ticket-agent`

新增：

- `AgentFacade`
- `AgentChatResult`

保留：

- `IntentRouter`
- `TicketAgentOrchestrator`
- `AgentExecutionGuard`
- `AgentToolRegistry`
- `AgentToolExecutor`
- `SpringAiLlmClient`

删除：

- `AgentChatController`
- `AgentChatService`

原因：

- HTTP 入口应该归属 `smart-ticket-api`。
- `smart-ticket-agent` 保留对话与 Tool 编排能力，不直接承担 Web Controller 职责。

## 4. Spring AI ChatClient 接入方式

`AgentFacade` 显式注入：

```text
ObjectProvider<ChatClient>
AgentLlmProperties
```

当前处理方式：

- `AgentFacade` 判断 `ChatClient` 是否可用，并在响应中返回 `springAiChatReady`。
- 实际模型调用仍由 `LlmAgentService -> LlmClient -> SpringAiLlmClient -> ChatClient` 完成。
- 这样可以避免一次请求内重复调用模型，同时满足 Agent 入口对 Spring AI 能力的接入。

启用条件：

```text
SMART_TICKET_AI_CHAT_ENABLED=true
SPRING_AI_MODEL_CHAT=openai
OPENAI_API_KEY=<your-api-key>
```

未启用时：

- `springAiChatReady=false`
- Agent 继续走规则 fallback 和现有受控 Tool 编排链路

## 5. 边界说明

Agent 入口不直接执行业务写入。

写操作仍必须经过：

```text
Agent Tool -> TicketService -> biz 权限/状态/幂等/日志
```

RAG 仍只在历史经验检索和创建前相似案例参考中使用，不替代当前工单事实查询。
