# Knowledge Build Change 5

更新时间：2026-04-20

## 1. 本次变更目标

本次在 `smart-ticket-biz` 中明确并整理 `TicketKnowledgeService`。

它的定位是：

```text
知识生产层
```

不是：

```text
向量化层
检索层
工单关闭主流程
Spring AI 调用层
```

## 2. TicketKnowledgeService 职责

实现位置：

```text
smart-ticket-biz/src/main/java/com/smartticket/biz/service/TicketKnowledgeService.java
```

职责包括：

- 判断工单是否具备知识沉淀条件。
- 读取 `ticket` 当前事实数据。
- 读取 `ticket_comment` 协作过程数据。
- 提取关键评论。
- 生成标准化知识正文。
- 生成知识摘要。
- 写入或更新 `ticket_knowledge`。

当前沉淀条件：

```text
工单状态 = CLOSED
并且
存在解决摘要 或 存在关键评论
```

这样可以避免把未完成、未验证或缺少处理信息的问题写入知识库。

## 3. 为什么放在 biz

`TicketKnowledgeService` 放在 `biz`，因为它需要理解工单业务事实和业务准入条件。

它依赖的问题是业务问题：

- 工单是否已经关闭？
- 关闭后的工单是否有沉淀价值？
- 哪些评论可以代表处理过程？
- 解决摘要和问题描述应该如何组织成业务知识？
- 同一来源工单的知识记录应该新增还是更新？

这些判断都属于工单业务语义，不属于模型能力，也不属于基础设施能力。

因此边界是：

```text
biz: 生产标准化知识文本
rag: 对知识文本切片、embedding、检索
infra: 提供 Redis、EmbeddingModel、VectorStore、PGvector 等外部适配
```

## 4. 为什么不能放在 rag

`rag` 的职责是知识向量化和检索，不应该决定业务准入。

如果把 `TicketKnowledgeService` 放到 `rag`，会导致：

- RAG 模块需要理解工单状态机。
- RAG 模块需要读取和解释工单评论类型。
- RAG 模块会参与判断工单是否可沉淀。
- 知识生产和向量检索边界混在一起。

这会让 `rag` 从“检索能力层”变成“业务规则层”，破坏模块边界。

正确关系：

```text
TicketKnowledgeService -> 写入 ticket_knowledge
EmbeddingService -> 读取 ticket_knowledge 并向量化
RetrievalService -> 检索已向量化知识
```

## 5. 为什么不能放在 infra

`infra` 只负责外部能力适配。

它可以承接：

- Redis 客户端
- Spring AI ChatClient 配置
- Spring AI EmbeddingModel 配置
- PGvector / VectorStore 配置
- 第三方 SDK 适配

它不应该承接：

- 工单是否可沉淀的判断
- 工单评论如何筛选
- 业务知识正文如何组织
- `ticket_knowledge` 的业务写入语义

如果把知识生产放到 `infra`，会导致基础设施层反向理解业务领域，形成错误依赖。

## 6. 与 Spring AI 的边界

`TicketKnowledgeService` 当前不调用 Spring AI。

它不做：

- ChatClient 调用
- LLM 摘要
- EmbeddingModel 调用
- VectorStore 写入
- 向量检索
- 相似案例召回

后续如果引入 LLM 辅助摘要，也必须遵守：

- biz 决定是否可沉淀。
- LLM 只能辅助生成文本草稿。
- 最终写入的业务准入和字段结构仍由 biz 控制。
- 向量化仍由 rag 承接。

## 7. 与工单关闭主流程的边界

`TicketKnowledgeService` 不承担工单关闭主流程。

关闭主流程属于：

```text
TicketService.closeTicket()
```

关闭主流程负责：

- 权限校验
- 状态从 `RESOLVED` 流转到 `CLOSED`
- 写操作日志
- 缓存失效
- 事务提交后发布关闭事件

知识构建是关闭成功后的后置动作。

推荐链路：

```text
TicketService.closeTicket()
  -> 主事务提交
  -> 发布 TicketClosedEvent
  -> 异步监听
  -> TicketKnowledgeService.buildKnowledge(ticketId)
  -> 写入 ticket_knowledge
  -> 后续由 rag 进行切片和向量化
```

知识构建失败不应影响工单关闭成功。

## 8. 知识文本模板

当前 `TicketKnowledgeService` 生成的正文模板：

```markdown
# 工单知识

## 基本信息
工单ID：{ticket.id}
工单编号：{ticket.ticketNo}
标题：{ticket.title}
分类：{ticket.category}
优先级：{ticket.priority}
状态：{ticket.status}
提单人ID：{ticket.creatorId}
处理人ID：{ticket.assigneeId}
创建时间：{ticket.createdAt}

## 问题描述
{ticket.description}

## 解决摘要
{ticket.solutionSummary 或 "暂无解决摘要"}

## 关键评论
- [{comment.commentType}] {comment.content}
```

摘要模板：

```text
{title}；分类={category}；优先级={priority}；解决={solutionSummary}
```

如果没有解决摘要但存在关键评论：

```text
{title}；分类={category}；优先级={priority}；关键处理={firstKeyComment.content}
```

摘要最大长度：

```text
900
```

## 9. 关键评论提取策略

第一版使用可解释的规则排序，不调用 LLM：

```text
SOLUTION      -> 优先级 0
PROCESS_LOG   -> 优先级 1
其他有内容评论 -> 优先级 2
```

同一优先级内按评论创建时间排序。

最多保留：

```text
8 条
```

## 10. 明确禁止

`TicketKnowledgeService` 中禁止：

- 直接调用向量检索。
- 直接调用 Spring AI EmbeddingModel。
- 直接写 `ticket_knowledge_embedding`。
- 直接访问 PGvector。
- 承担关闭工单主流程。
- 替代 `TicketService.closeTicket()` 做状态流转。
