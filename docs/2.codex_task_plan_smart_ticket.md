# Codex 分阶段任务清单（企业智能工单协同平台，最新版）

## 1. 文档用途

这份文档用于把“企业智能工单协同平台”项目拆成适合直接发给 Codex 的阶段性任务。

本版任务清单基于当前项目最新确认内容进行更新，重点强调：

1. **项目本体是后端业务系统，Agent 是增强层**
2. **第一版只做模块化单体，不做微服务**
3. **不采用重 DDD 架构**
4. **暂不引入 MCP、LLM Wiki、复杂多 Agent 并行；LLM 在 Tool 标准化后接入**
5. **先完成真实业务闭环，再增加受控 Agent 编排与 RAG 能力**
6. **严格区分事实查询、业务执行、历史经验检索三类请求**
7. **工单关闭后异步进行知识构建与向量入库**
8. **LLM 只参与理解、规划和总结，业务执行必须通过 Tool 和 biz 层**

---

## 2. Codex 使用原则

1. 一次只发一个阶段任务，不要把整份文档一次性丢给 Codex。
2. 每个阶段完成后，先本地运行、看 diff、调接口、改命名，再进入下一阶段。
3. 每次发任务时，都要求 Codex：
   - 说明做了哪些修改
   - 列出新增/修改的文件
   - 给出运行与验证方式
   - 不要擅自引入超出当前阶段范围的复杂设计
4. 所有业务写操作必须走 `smart-ticket-biz`，Agent 不能绕过业务层。
5. RAG 只用于历史经验检索，不用于当前事实查询。
6. 创建工单链路可以加入“创建前相似案例检查”，但不能强制阻止用户提单。
7. LLM 接入后也不能直接执行写操作，必须先生成受控 Tool 调用计划，再由代码校验并调用 Tool。

## 3. 项目背景（建议先放入 README 和 docs/dev-guide.md）

项目名称：**企业智能工单协同平台**

项目定位：

- 作为 **Java 后端实习项目**：体现业务建模、权限控制、状态流转、缓存、幂等、日志、测试、部署能力
- 作为 **Agent 开发实习项目**：体现意图路由、Tool Calling、RAG、会话上下文、自然语言业务入口、Agent 与业务系统结合能力

技术栈建议：

- Java 17
- Spring Boot 3.x
- Spring Security + JWT
- MySQL
- Redis
- PostgreSQL + pgvector
- Spring AI
- Docker Compose
- JUnit 5 + Mockito

模块设计：

```text
smart-ticket-platform
├─ smart-ticket-app
├─ smart-ticket-common
├─ smart-ticket-auth
├─ smart-ticket-domain
├─ smart-ticket-biz
├─ smart-ticket-agent
├─ smart-ticket-rag
├─ smart-ticket-infra
```

- `common`：通用基础能力

- `rag`：历史案例知识构建后的向量化与检索
- `infra`：Redis、LLM、Embedding、pgvector、文件存储等适配
- `api`：Controller、DTO、VO、接口层

## 4. 项目边界与实现纪律

### 4.1 Agent 边界

Agent 负责：

- 意图识别
- 信息澄清
- 参数整理
- Tool 调用编排
- 历史案例检索触发
- 返回结果摘要

Agent 不负责：

- 直接写数据库
- 跳过权限校验
- 事务一致性
- 业务规则裁决
- 默认每次都走 RAG

Agent 内部建议按三层职责推进：

- LLM 层：负责意图识别、参数抽取、缺参澄清、轻量规划和结果总结。
- Tool 层：负责把可执行能力封装成受控工具，声明输入输出、风险等级、是否只读、是否需要用户确认。
- biz 层：负责最终业务规则、权限校验、状态机、事务一致性、操作日志和事实状态。

后续开发必须保持这条边界：LLM 可以建议调用哪个 Tool，但不能绕过 Tool 和 `biz` 直接改变工单状态。

### 4.2 RAG 边界

RAG 只用于：

- 历史工单经验检索
- 相似案例召回
- 创建前相似案例检查
- 经验参考输出

RAG 不用于：

- 当前工单状态查询
- 当前负责人查询
- 当前事实数据判断
- 工单写操作

### 4.3 模块边界

- `api` 不直接访问 `domain`
- `agent` 不绕过 `biz`
- `rag` 不参与主事务写入
- `auth` 负责“你是谁”，`biz` 负责“你能不能操作这张工单”
- `common` 只放真正跨模块复用的公共能力
- `TicketKnowledgeService` 放在 `biz`
- `EmbeddingService` 与 `RetrievalService` 放在 `rag`

## 14. 阶段 8：LLM 接入与 Prompt 层

### 目标
接入大模型能力，但先只让 LLM 做理解和表达，不让 LLM 直接执行写操作。

阶段 8 的准确定位是“LLM 增强的固定工作流”，还不是完整 Agent 编排。当前流程仍由代码固定控制：规则路由兜底、LLM 辅助识别意图、规则参数兜底、LLM 辅助抽取参数、代码选择 Tool、Tool 调用 biz、LLM 辅助总结。

本阶段不要求 LLM 自主规划流程，不要求 LLM 主动选择多个 Tool，也不要求形成“计划 -> 行动 -> 观察 -> 再决策”的循环。这些内容放到阶段 9。

### 范围

## 16. 阶段 10：会话上下文增强与指代消解

### 目标
让 Agent 能处理“它、刚才那个工单”这类上下文，并支持缺参追问后的继续执行。

阶段 10 基于阶段 9 的单 Agent 编排结构继续推进。当前 `TicketAgentOrchestrator` 已经负责主编排，`AgentContextUpdater` 已经集中维护会话上下文，因此本阶段重点不是继续扩展 LLM 计划能力，而是补齐多轮对话状态。

阶段 10 的核心是 pending action：当 Tool 因缺少必要参数返回 `NEED_MORE_INFO` 时，将当前计划、已知参数和缺失字段保存到会话上下文；用户下一轮补充信息后，优先恢复 pending action，合并新参数，再继续校验和执行。

### 发给 Codex 的任务

```text
请实现 Agent 会话上下文增强。

要求：
1. 基于 Redis 中的 agent:session:{sessionId} 保存会话上下文
2. 至少维护以下字段：
   - activeTicketId
   - activeAssigneeId
   - lastIntent
   - recentMessages
3. 增加 pending action 结构，建议包含：
   - pendingIntent
   - pendingToolName
   - pendingParameters
   - awaitingFields
   - lastToolResult
4. 在用户发起新消息时，先判断是否存在 pending action
5. 如果存在 pending action：
   - 从当前消息抽取补充参数
   - 与 pendingParameters 合并
   - 重新校验 requiredFields
   - 参数完整后继续通过 AgentToolExecutor 执行
   - 执行成功或失败后清理 pending action
6. 如果不存在 pending action，再走阶段 9 的正常单 Agent 编排流程
7. 支持简单指代消解：
   - “它”“这个”“刚才那个工单”优先映射到 activeTicketId
   - “他”“这个处理人”优先映射到 activeAssigneeId
5. 支持以下场景：
   - “帮我查一下刚才那个工单”
   - “把它转给张三”
   - “就按刚才那个创建”
   - “补充一下，优先级是高”
8. 在 docs/learning/stage-10-conversation-context.md 中写清楚上下文结构、更新时机和缺参追问流程
9. 为生成的类、字段、方法等写清楚中文注释

限制：
- 第一版不做复杂长期记忆
- 不做过度复杂的实体图谱
- 不做多 Agent
- 不做多 Tool 自动循环
```

### 验收标准
- 多轮对话能引用上一轮工单
- 缺参追问后能继续执行 pending action
- 结构简单但可解释

### 当前实现状态
- 已在 `AgentSessionContext` 中补充 `pendingAction`，继续通过 Redis key `agent:session:{sessionId}` 保存会话上下文。
- 已新增 `AgentPendingAction`，包含 `pendingIntent`、`pendingToolName`、`pendingParameters`、`awaitingFields`、`lastToolResult`。
- `TicketAgentOrchestrator` 读取上下文后会优先判断 pending action；存在 pending action 时先恢复补参流程，不重新开始完整路由。
- `AgentPendingActionService` 负责抽取本轮补充参数、合并 pending 参数、重新校验计划并继续通过 `AgentToolExecutor` 执行。
- `AgentContextReferenceResolver` 负责简单指代消解，把“它”“这个”“刚才那个工单”映射到 `activeTicketId`，把“他”“这个处理人”映射到 `activeAssigneeId`。
- Tool 返回 `NEED_MORE_INFO` 时保存或刷新 pending action；返回 `SUCCESS` 或 `FAILED` 时清理 pending action。
- 阶段十说明文档已补充到 `docs/learning/stage-10-conversation-context.md`。

### 当前边界
- 当前不会做长期记忆、复杂实体图谱、多 Agent 或多 Tool 自动循环。
- “把它转给张三”这类表达目前可以完成工单指代消解，但处理人姓名到用户 ID 的解析还没有作为独立能力落地；如果缺少 `assigneeId`，系统会继续追问目标处理人 ID。
- RAG 仍未默认接入 Agent 主链路，后续阶段只应在历史经验检索类场景中受控接入。

### 阶段 10.5：执行边界平台化整理

在进入阶段 11 前，建议做一次小重构，把执行前校验从编排流程中收拢为平台能力。

已完成整理：
- 新增 `AgentExecutionGuard`，统一处理 Tool 是否存在、Tool 是否支持 intent、风险等级、用户确认和 requiredFields 校验。
- 新增 `AgentExecutionDecision` 和 `AgentExecutionDecisionStatus`，把执行前结果归一为 `ALLOW_EXECUTE`、`NEED_CONFIRMATION`、`NEED_MORE_INFO`、`REJECTED`。
- `TicketAgentOrchestrator` 不再直接关心确认和缺参细节，只消费 Guard 决策并串联流程。
- `AgentPendingActionService` 恢复 pending action 时也复用同一个 Guard，避免正常流程和补参流程各写一套安全判断。
- `AgentToolExecutor` 收敛为纯执行器，只调用已经被 Guard 放行的 Tool。

整理后的职责边界：

```text
LLM：理解、计划、总结
TicketAgentOrchestrator：编排流程
AgentExecutionGuard：执行前策略和安全边界
AgentToolExecutor：调用 Tool
AgentTool：封装业务能力
biz：最终权限、状态机、事务和并发裁决
```

```text
9. LLM 只能参与理解、规划和总结，业务执行必须通过 Tool 和 biz 层
```

## 23. 给你的实际使用建议

最稳的使用方式：

1. 每次只复制一个阶段任务给 Codex
2. Codex 改完后，你自己：
   - 看 diff
   - 跑启动
   - 调接口
   - 改命名和注释
   - 记录设计理由
3. 验收通过后，再进入下一个阶段

不要一次性把所有阶段都交给 Codex。  
你的目标不是“最快生成项目”，而是：

> **做出一个你自己能讲清楚、能写进简历的项目**
